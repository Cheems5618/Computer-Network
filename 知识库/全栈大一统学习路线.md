# 方案二：“大一统”全栈网络学习路线图

既然决定三个方向都学，我们可以将它们串联成一个**由底向上，再由微观到宏观**的终极学习路线。
学习顺序建议为：**硬核推演（奠基） -> 源码探究（应用） -> 架构设计（升华）**。

## 模块一：筑基阶段（自底向上的硬核推演）
**目标：建立无可挑剔的网络底层常识，不再惧怕任何 IP 和子网问题。**
1. **IP 协议与 CIDR 魔法**
   - 彻底搞懂 `192.168.0.0/16` 这种写法：如何用二进制按位与计算来判断 IP 归属。
   - 实践：手写一段 Python 代码，模拟路由器判断 `192.168.5.9` 是否在 `/16` 网段内。
2. **端口与本地环回机制**
   - 深挖 `127.0.0.1`（Loopback）的工作机制：为什么拔掉网线它依然能通？数据包是如何在内核协议栈中“折返”的。
3. **TCP/UDP 与连接建立**
   - 掌握三次握手，理解“端口号”在操作系统层面对应的是什么数据结构（Socket）。

## 模块二：破局阶段（基于问题的源码级探究）
**目标：通过真实工具（curl, clash）理解理论在代码中是如何落地的。**
1. **`curl` 请求全生命周期拆解**
   - 沿着排错记录，模拟一次 `curl -v http://192.168.5.9:3000`。
   - 探究：环境变量 `http_proxy` 是如何被 `curl` 进程读取并改变其网络请求目标的。
2. **正向代理与 SOCKS/HTTP 代理协议**
   - 了解当你访问网页时，Clash 是如何作为“中间人”接管流量的。
   - 抓包实践：用 Wireshark 抓取代理前和代理后的 HTTP CONNECT 报文对比。
3. **版本代沟导致的 Bug 分析**
   - 探究为何旧版 curl 无法解析 CIDR，这涉及到 C 语言中对 IP 字符串进行 `inet_pton` 和位运算的实现差异。

## 模块三：造物阶段（宏观架构与流量拓扑设计）
**目标：跳出单一机器，站在集群架构师的角度规划流量。**
1. **网络地址转换 (NAT) 与内网穿透**
   - 从局域网 IP 到公网 IP 的跃迁。代理服务器在云端是如何帮你转发流量的。
2. **TUN 虚拟网卡的高阶应用**
   - 不依赖环境变量的终极杀器：操作系统级别接管流量。Clash 的 TUN 模式是如何修改系统路由表（Routing Table）的。
3. **高性能分流引擎的底层数据结构**
   - Clash 为何能瞬间匹配数万条规则？学习前缀树（Trie Tree）在 IP CIDR 匹配和域名匹配（SNI 嗅探）中的应用。
